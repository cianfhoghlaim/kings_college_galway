## Detailed Findings

### 1. Cloudflare ✅

**Status:** Official OpenAPI 3.0.3 specification available

**Access:**
- YAML: `https://raw.githubusercontent.com/cloudflare/api-schemas/main/openapi.yaml`
- JSON: `https://raw.githubusercontent.com/cloudflare/api-schemas/main/openapi.json`
- Repository: https://github.com/cloudflare/api-schemas

**Coverage:**
- 600+ API endpoints
- 20+ service categories
- AI/ML, DNS, Security, Workers, Storage, etc.

**Key Details:**
- Migrated from JSON Hyper-Schema in Nov 2022
- Used for Terraform provider generation
- Large file size (~10+ MB)
- BSD-3-Clause licensed

**Documentation:** See `cloudflare/cloudflare-openapi-specification-research.md`

### 2. Crawl4AI ✅

**Status:** Dynamically generated OpenAPI specification

**Access:**
- Local: `http://localhost:11235/openapi.json` (requires running Docker container)
- Swagger UI: `http://localhost:11235/docs`
- Third-party: https://apify.com/janbuchar/crawl4ai/api/openapi

**Quick Start:**
```bash
docker pull unclecode/crawl4ai:latest
docker run -d -p 11235:11235 --name crawl4ai --shm-size=1g unclecode/crawl4ai:latest
curl http://localhost:11235/openapi.json > crawl4ai-openapi.json
```

**Coverage:**
- Core crawling (sync, async, streaming)
- HTML/PDF/screenshot extraction
- LLM-powered extraction
- Job management with webhooks
- WebSocket and SSE endpoints

**Key Details:**
- FastAPI-based auto-generation
- JWT authentication
- Redis-backed task management
- Prometheus metrics

**Documentation Saved:** `/home/user/hackathon/research/api-specs/crawl4ai-openapi-research.md`

### 4. Komodo (komo.do) ❌

**Status:** No OpenAPI specification (by design)

**Alternative:** Typeshare-based TypeScript/Rust client generation

**Access:**
- TypeScript: `npm install komodo_client`
- Rust: `cargo add komodo_client`
- Documentation: https://docs.rs/komodo_client/latest/komodo_client/

**API Style:** RPC-like HTTP with JSON serialization

**Endpoints:**
- `/auth`, `/user`, `/read`, `/write`, `/execute`, `/terminal`

**Key Details:**
- Rust-first development approach
- Automatic TypeScript type generation from Rust
- Single source of truth (no spec drift)
- Could add `utoipa` for OpenAPI generation if needed

**Documentation Saved:**
- `/home/user/hackathon/research/api-specs/komodo-openapi-research.md`
- `/home/user/hackathon/research/api-specs/komodo-api-summary.md`

### 5. Pangolin ✅

**Status:** Official OpenAPI 3.x specification (dynamically generated)

**Access:**
- Cloud: https://api.pangolin.net/v1/docs
- Self-hosted: `https://api.{your-domain}/v1/docs`
- OpenAPI JSON: Download from Swagger UI

**Coverage:**

- Identity & access control
- Monitoring & observability

**Key Details:**
- Integration API introduced in v1.3.0 (Oct 2024)
- GA in v1.4.0 (Nov 2024)
- Not in GitHub repo (in .gitignore)
- Must download from running instance

**Client Generation:**
```bash
npx swagger-typescript-api generate \
  --path openapi/pangolin-api.json \
  --output src/pangolinClient/ \
  --axios
```

### 6. Gradio ✅

**Status:** Auto-generated OpenAPI v3 for every Gradio app

**Access:**
- Any Gradio app: `https://<your-app-url>/gradio_api/openapi.json`
- Swagger UI alternative: `?view=api`
- Info endpoint: `/info?all_endpoints=true`

**Dual Functionality:**
- **Produces:** Every app exposes its own spec
- **Consumes:** `gr.load_openapi()` generates UIs from external specs

**Key Details:**
- Zero-config automatic generation
- FastAPI + Pydantic based
- Complete with types, parameters, examples
- Compatible with standard OpenAPI tooling

**Use Cases:**
- API exploration and testing
- Client SDK generation
- MCP server deployment
- Interactive API documentation

**Documentation Saved:** `/home/user/hackathon/research/shared/gradio-openapi-research.md`

### 10. Dagster ❌

**Status:** No OpenAPI specification (GraphQL-first design)

**Primary API:** GraphQL at `/graphql`
- Schema: `./js_modules/dagster-ui/packages/ui-core/src/graphql/schema.graphql`
- Interactive playground available

**Limited REST API:** Only 3 endpoints
- `/report_asset_materialization/{ASSET_KEY}`
- `/report_asset_check/{ASSET_KEY}`
- `/report_asset_observation/{ASSET_KEY}`

**Alternative Approaches:**
- Use GraphQL API (recommended)
- Use Python SDK (`dagster-graphql`)
- Convert GraphQL schema to OpenAPI (limited usefulness)

**Key Details:**
- GraphQL schema evolving, subject to breaking changes
- Python-native development philosophy
- REST API only for external asset reporting
